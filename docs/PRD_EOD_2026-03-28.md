## 2. Product Summary

GREY-EARTH is a SIGINT-TERRAIN-oriented geospatial analysis surface built for low-relief tactical terrain interpretation, terrain interrogation, profile generation, and future line-of-sight / operational overlay workflows.

The intended product is not a generic GIS map.

The intended product is:

- a doctrine-driven terrain interface
- a deterministic render stack
- a map + HUD + analysis product system
- a deployable briefing and operational awareness surface
- a bridge between terrain physics, operator interpretation, and future mission-analysis overlays

The current codebase now materially reflects that direction.

## 3. Product Goal

### Primary Goal

Deliver a production-ready tactical terrain interface that:

- renders honest terrain relief for the NYC littoral operating area
- supports rapid elevation interrogation and transect analysis
- exposes a manifest-driven render model rather than ad hoc layer state
- supports both 2D and 3D terrain modes
- uses real remotely sourced imagery instead of placeholder substrate only
- is deployable on Vercel with browser-based Earth Engine authentication

### Secondary Goal

Create a code architecture that can absorb future features without another major rewrite, including:

- contours
- slope analysis
- hypsometric tinting
- line-of-sight
- Fresnel analysis
- more explicit denied / threat geometry
- terrain verification / rendering package outputs aligned to the local SIGINT terrain bundle

## 5. What We Went Through

### 5.1 Discovery And Reality Check

The first step was a repo and build audit. That established:

- the application was already beyond the original PRD’s “current state”
- the codebase had real terrain analysis hooks and a working 3D component
- the repo was not initially a git repository at the workspace root
- lint quality and architecture quality were behind feature ambition

### 5.2 Structural Refactor

The monolithic map/HUD coupling was replaced with subsystem-first architecture:

- viewport state
- render state
- analysis state
- map runtime
- Earth Engine runtime

This refactor moved the app toward a manifest-first rendering model and away from direct panel-to-map mutation.

### 5.3 Visual Doctrine Pass

The product moved away from placeholder purple-ish and generic substrate styling and toward the local tactical dark-relief doctrine:

- onyx substrate
- hillshade grain
- confidence stipple
- denied-zone crosshatch
- scan-line and aging wash treatment

### 5.4 Controlled Terrain Base

The remaining external substrate dependency was removed so the terrain stack was driven by internal treatment rather than leaning on the prior base style.

This created a cleaner visual system, but it also made the absence of authenticated real imagery more obvious.

### 5.5 Real Earth Engine Path

The system was then extended to support:

- browser-based Earth Engine authentication
- Earth Engine tile URL derivation
- a real imagery raster layer inserted into the MapLibre stack
- production deployment with public Earth Engine project/client configuration

### 5.6 Usability Adjustment

The first Earth Engine activation path existed only as a small HUD panel.

That was too easy to miss, so a larger activation overlay was added in the main viewport to make the sign-in action explicit.

## 8. Current Architecture

### 8.1 Top-Level App Shell

The app is now composed through dedicated providers rather than one monolithic map context.

Active providers:

- `ViewportStateProvider`
- `EarthEngineProvider`
- `AnalysisStateProvider`
- `RenderStateProvider`
- `MapRuntimeProvider`

This is the most important structural improvement made during the sprint.

### 8.2 State Domains

#### Viewport State

Owns:

- center
- zoom
- pitch
- bearing
- breakpoint
- device class
- current view mode (`2D` / `3D`)

#### Render State

Owns:

- requested layers
- active layer IDs
- suppressed layer IDs
- terrain profile
- warnings
- manifest output

#### Analysis State

Owns:

- active tool
- transect points
- profile data
- selected query point
- analysis status

#### Map Runtime

Owns:

- imperative `maplibre-gl` map reference
- map load state

#### Earth Engine Runtime

Owns:

- auth status
- project ID
- imagery tile template
- imagery label
- runtime error state

### 8.3 Declarative Rendering Direction

The current render model is doctrine-aware and manifest-oriented.

Implemented concepts include:

- canonical layer order
- device-class degradation
- terrain profile inference
- suppressed-layer reasoning
- analysis product flags

This is aligned with the local `sigint_terrain_bundle` direction even though full package generation is not yet implemented.

## 12. What Is Operationally Risky

### 12.1 Earth Engine Activation Risk

The most immediate operational risk is not code quality, but browser auth and Google Cloud configuration.

Failure modes:

- popup blocker prevents sign-in
- allowed JavaScript origins do not match deployed domains
- account lacks Earth Engine access
- Earth Engine auth succeeds but imagery call is quota-limited or denied

### 12.2 Honest Layer Disclosure

The current system intentionally exposes unsupported layers in the doctrine model while suppressing them in the active render stack.

This is correct behavior, but product messaging must remain honest so operators do not assume contour/slope/hypso/LOS are live.

### 12.3 3D Fidelity Gap

The 3D renderer works, but it should still be treated as an analysis/briefing prototype rather than finished mission-grade terrain presentation.

### 12.4 Data Provenance Gap

The app does not yet surface full source provenance, freshness, and analysis confidence in the way a fielded operational tool should.

## 14. Requirements For The Next Phase

### 14.1 Earth Engine Must Become Routine, Not Optional

Requirements:

- successful auth on live production
- Earth Engine imagery as the default real terrain base after sign-in
- clear operator messaging when auth is absent or blocked
- no hidden silent failure states

### 14.2 Manifest Must Become Exportable

Requirements:

- persist current render state manifest to JSON
- generate explicit suppressed-layer reasons
- prepare path for verification report generation

### 14.3 Terrain Overlay Roadmap Must Become Real

Requirements:

- contours derived from DEM
- slope overlay from DEM processing
- hypsometric palette from doctrine rules
- LOS first implementation from analysis points

### 14.4 3D Must Mature

Requirements:

- improved terrain texture composition
- better elevation tiling strategy
- stronger viewport synchronization with 2D context
- future support for operational overlays in 3D mode
